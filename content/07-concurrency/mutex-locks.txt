/// 简介

上节创建了线程，但多个线程访问共享数据时会产生**数据竞争（Data Race）**——这是未定义行为。**互斥量（Mutex, Mutual Exclusion）**是最基本的同步原语：确保同一时刻只有一个线程进入**临界区（Critical Section）**。C++ 提供了从简单到灵活的多级锁守卫，以及 C++17 的 `scoped_lock` 自动防死锁机制。

本节回答两个核心问题：**用什么锁？** 以及 **怎么避免死锁？**

/// 知识点列表

\list{
- std::mutex —— lock/unlock 与 RAII 守卫
- std::lock_guard —— 最简 RAII 锁
- std::unique_lock —— 可转移、可延迟、可中途释放
- std::scoped_lock（C++17）—— 多锁原子获取，自动防死锁
- std::shared_mutex / shared_lock（C++17）—— 读写锁
- std::recursive_mutex —— 可重入锁（慎用）
- 死锁的四个必要条件与预防策略
- 锁粒度选择与性能权衡
}

/// 知识点详解

// 数据竞争——为什么需要锁

没有同步时，两个线程同时读写同一变量是**未定义行为（UB）**，即使你"不在乎"结果的值。

\run[cpp]{
#include <iostream>
#include <thread>

int counter = 0;

void increment() {
    for (int i = 0; i < 100000; ++i)
        counter++; // 非原子操作: 读→加→写 三步可被中断
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);
    t1.join();
    t2.join();
    std::cout << "counter = " << counter
              << " (expected 200000)\n";
    return 0;
}
}

运行多次，`counter` 几乎不可能等于 200000——这就是**数据竞争**。解决方案：用互斥量保护共享访问。

// std::mutex

\code[cpp]{
#include <mutex>

std::mutex mtx;
int counter = 0;

void safeIncrement() {
    for (int i = 0; i < 100000; ++i) {
        mtx.lock();   // 获取锁——若已被占用则阻塞
        counter++;     // 同一时刻只有一个线程执行此处
        mtx.unlock(); // 释放锁
    }
}
}

\danger{**永远不要直接调用 `lock()`/`unlock()`。** lock 和 unlock 之间若抛出异常，unlock 不会执行——锁永远不被释放→死锁。必须使用 RAII 守卫。}

// RAII 锁守卫——lock_guard / unique_lock / scoped_lock

C++ 用 RAII 解决"忘记 unlock"问题：构造时加锁，析构时解锁，异常安全。

\tabs{
\tab[lock_guard]{
\code[cpp]{
{
    std::lock_guard<std::mutex> guard(mtx); // 构造→lock
    counter++;
} // 析构→unlock，异常也不漏
// C++17 CTAD 简写：std::lock_guard guard(mtx);
}
**特点**：最简单，零开销。不能中途 unlock，不能转移。
}
\tab[unique_lock]{
\code[cpp]{
{
    std::unique_lock lock(mtx);    // 构造→lock
    counter++;
    lock.unlock();                  // 提前释放
    // ... 不需要锁的计算 ...
    lock.lock();                    // 重新获取
    counter++;
} // 析构→若持有锁则 unlock
}
**特点**：可中途解/重锁、可延迟加锁（`std::defer_lock`）、可转移所有权。条件变量必须搭配它。
}
\tab[scoped_lock (C++17)]{
\code[cpp]{
std::mutex mtx1, mtx2;
{
    std::scoped_lock lock(mtx1, mtx2); // 原子获取两把锁
    // 操作 mtx1 和 mtx2 保护的数据
} // 同时释放
}
**特点**：接受任意数量的 mutex，内部用 `std::lock` 算法避免死锁。多锁场景首选。
}
}

\table{
守卫类型 | 可延迟 | 可中途 unlock | 可转移 | 多锁 | 适用场景
---
`lock_guard` | ✗ | ✗ | ✗ | ✗ | 简单临界区
`unique_lock` | ✓ | ✓ | ✓ | ✗ | 条件变量、延迟锁定
`scoped_lock` | ✗ | ✗ | ✗ | ✓ | 同时锁多个 mutex
}

\quiz[choice|单选]{
Q: 你需要在等待条件变量时使用哪种锁守卫？
A: std::lock_guard
B: std::unique_lock
C: std::scoped_lock
D: 直接调用 mtx.lock()
ANS: B
EXP: `condition_variable::wait()` 需要在等待期间释放并重新获取锁，只有 `unique_lock` 提供了这种灵活性。`lock_guard` 和 `scoped_lock` 不支持中途 unlock/relock。
}

// std::shared_mutex / shared_lock（读写锁, C++17）

当读操作远多于写操作时（如配置缓存、游戏世界状态查询），**读写锁（Reader-Writer Lock）**允许多个读者并发，只有写者独占。

\code[cpp]{
#include <shared_mutex>
#include <unordered_map>

class ThreadSafeCache {
    mutable std::shared_mutex mtx_;
    std::unordered_map<std::string, int> data_;
public:
    int get(const std::string& key) const {
        std::shared_lock lock(mtx_);    // 共享锁——多线程可同时持有
        auto it = data_.find(key);
        return it != data_.end() ? it->second : -1;
    }

    void set(const std::string& key, int val) {
        std::unique_lock lock(mtx_);    // 独占锁——排斥所有其他锁
        data_[key] = val;
    }
};
}

\warning{**`shared_mutex` 不是万能药。** 如果写操作频繁，读者和写者频繁切换锁模式反而比普通 `mutex` 更慢（shared_mutex 的加锁成本更高）。只在读写比 > 10:1 时使用。}

// std::recursive_mutex

允许同一线程多次加锁（加多少次就要解多少次）。

\code[cpp]{
std::recursive_mutex rmtx;

void funcA() {
    std::lock_guard lock(rmtx);
    funcB(); // funcB 也加 rmtx，同一线程不会死锁
}
void funcB() {
    std::lock_guard lock(rmtx);
    // ...
}
}

\danger{**需要 `recursive_mutex` 通常意味着设计缺陷。** 它掩盖了锁的滥用——应该重构为：将需要锁的操作提到统一的入口函数，内部不再重复加锁。仅在无法修改遗留代码时考虑。}

// 死锁——四个必要条件与预防

\diagram[flow]{
线程A: lock(mtxA) -> 等待 mtxB
线程B: lock(mtxB) -> 等待 mtxA
结果: 循环等待 -> 死锁!
}

**死锁四个必要条件（Coffman Conditions）**：

\table{
条件 | 含义 | 打破方法
---
互斥（Mutual Exclusion） | 资源一次只能被一个线程持有 | 用无锁算法/原子操作
持有并等待（Hold and Wait） | 持有一个锁的同时等待另一个 | scoped_lock 原子获取所有锁
不可抢占（No Preemption） | 不能强制拿走别人的锁 | try_lock + 超时
循环等待（Circular Wait） | 线程间形成等待环 | 固定全局锁获取顺序
}

实践中最有效的两个策略：

\run[cpp]{
#include <iostream>
#include <thread>
#include <mutex>

std::mutex mtxA, mtxB;
int dataA = 0, dataB = 0;

// 策略1: std::scoped_lock 自动防死锁
void transferSafe(int amount) {
    std::scoped_lock lock(mtxA, mtxB);
    dataA -= amount;
    dataB += amount;
}

// 策略2: 固定全局锁序——始终先 A 后 B
void transferOrdered(int amount) {
    std::lock_guard lockA(mtxA);
    std::lock_guard lockB(mtxB);
    dataA -= amount;
    dataB += amount;
}

int main() {
    std::thread t1(transferSafe, 10);
    std::thread t2(transferSafe, 20);
    t1.join();
    t2.join();
    std::cout << "A=" << dataA << " B=" << dataB << "\n";
    return 0;
}
}

\tip{**实战首选 `std::scoped_lock`。** 它内部使用 `std::lock` 算法（try-and-back-off），不管你传入锁的顺序如何，都能安全获取。}

// 锁粒度选择

\table{
粒度 | 优点 | 缺点 | 场景
---
粗粒度（一个大锁） | 简单，不易死锁 | 并发度低 | 原型期、访问频率低
细粒度（每个数据独立锁） | 并发度高 | 复杂、容易死锁 | 性能敏感的热路径
无锁（atomic） | 最高吞吐 | 极难正确实现 | 只用于久经验证的模式
}

\note{**"从粗到细"原则**：先用一把大锁保证正确性，profiling 发现锁竞争是瓶颈后再细化。过早细化是并发编程中最常见的过度工程。}

\collapse[锁粒度实战示例]{
\code[cpp]{
// 粗粒度：整个 Map 一把锁
class CoarseMap {
    std::mutex mtx;
    std::unordered_map<int, int> data;
public:
    void put(int k, int v) { std::lock_guard l(mtx); data[k] = v; }
    int get(int k) { std::lock_guard l(mtx); return data[k]; }
};

// 细粒度：分桶锁
class StripedMap {
    static constexpr int N = 16;
    std::mutex mtxs[N];
    std::unordered_map<int, int> buckets[N];
    int bucket(int k) { return std::hash<int>{}(k) % N; }
public:
    void put(int k, int v) {
        int b = bucket(k);
        std::lock_guard l(mtxs[b]);
        buckets[b][k] = v;
    }
};
}
}

/// 工程实践

\list{
- 锁保护的是**数据**，不是代码——所有访问路径都必须加锁，包括"只读"路径
- 持有锁时不要调用用户回调或虚函数——回调可能加同一把锁→死锁
- `mutex` 不可拷贝不可移动——包含 mutex 的类需自定义或删除拷贝/移动
- 临界区越短越好：加锁→操作共享数据→立即 unlock，不在临界区做 I/O 或分配
- 多锁一律用 `scoped_lock`，单锁简单路径用 `lock_guard`，需要灵活性用 `unique_lock`
}

/// 常用方法速查

\table{
类型 | 特征 | 适用场景
---
`mutex` | 基本互斥锁 | 通用
`timed_mutex` | 支持 try_lock_for 超时 | 需要限时等待
`recursive_mutex` | 同线程可重入 | 遗留代码兼容（尽量避免）
`shared_mutex` | 读写锁 | 读多写少（C++17）
`lock_guard` | 最简 RAII | 简单加锁
`unique_lock` | 灵活 RAII | 条件变量、延迟/转移
`scoped_lock` | 多锁防死锁 | 同时获取多把锁（C++17）
}

/// 新版特性（C++17/20/23）

\compare{
old: C++11 提供 `mutex`、`lock_guard`、`unique_lock`。多锁需手动调用 `std::lock(mtxA, mtxB)` 再用 `adopt_lock` 构造守卫——样板代码多且易错。没有标准读写锁（C++14 有 `shared_timed_mutex`）。

new: C++17 引入 `scoped_lock`（可变参数模板一步到位）和 `shared_mutex`。C++17 CTAD 使锁声明更简洁。C++20 的 `jthread` 减少了"异常路径忘 join"导致的锁+线程交互死锁。
}

/// 难点易错点分析

\danger{**锁保护的是数据，不是代码。** 只在 `write` 路径加锁而忘了 `read` 路径——依然是数据竞争（UB），即使"只是读"。}

\danger{**持有锁时 join 需要同一锁的线程→死锁。** 典型模式：主线程持锁 → join 工作线程 → 工作线程需要同一锁。修复：join 前释放锁。}

\quiz[tf|判断]{
Q: `std::shared_mutex` 的 `shared_lock` 允许多个线程同时持有，因此不存在写饥饿（Writer Starvation）问题。
ANS: F
EXP: 如果大量读者不断获取 shared_lock，写者可能长时间无法获取独占锁，造成写饥饿。标准没有规定 shared_mutex 的公平策略，实际行为取决于实现。高写频率场景应评估是否需要自己实现公平队列锁。
}

🟡 **进阶：线程安全读写缓存**

实现 `ThreadSafeCache<K, V>`：用 `shared_mutex` + `shared_lock`/`unique_lock` 保护内部 `unordered_map`。提供 `get(key)` 和 `set(key, val)` 接口。测试：8 个读线程 + 2 个写线程并发访问，用 TSan 验证无数据竞争。

🔴 **挑战：分桶并发哈希表**

实现按桶分锁的**并发哈希表**：
- 每个桶一个 `shared_mutex`，读用 `shared_lock`，写用 `unique_lock`
- 支持 `get`、`put`、`remove`、`forEach`
- 桶数量可配置，测试不同桶数对吞吐量的影响
- 对比全局锁版本和分桶版本的性能差异

\collapse[分桶锁设计提示]{
1. 用 `std::hash<K>{}(key) % num_buckets` 确定桶号
2. 每个桶是一个 `std::shared_mutex` + `std::list<std::pair<K,V>>`（链表法解决冲突）
3. `forEach` 需要逐桶加 shared_lock，但不要同时持有所有桶的锁
4. 性能测试参考：4 核机器上 16 桶通常是甜点
}
