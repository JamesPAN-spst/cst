/// 简介

本项目综合运用构建系统板块全部知识——编译模型理解、现代 CMake 配置、工具链集成——从零搭建一个专业级 C++ 项目骨架。完成后，你将拥有一套可复用的项目模板，作为后续所有 C++ 项目的起点。

/// 项目需求

**项目名称：ProjectForge — 专业 C++ 项目模板**

从空目录开始，搭建一个包含库、可执行文件、单元测试和 CI 配置的完整 CMake 项目。

\code[text]{
ProjectForge/
├── CMakeLists.txt              # 顶层 CMake 配置
├── CMakePresets.json            # 构建预设（debug/release/asan/tsan）
├── .clang-format                # 代码格式化配置
├── .clang-tidy                  # 静态分析配置
├── .gitignore
├── README.md                    # 构建说明和项目结构
├── src/                         # 库源码
│   ├── CMakeLists.txt
│   ├── include/
│   │   └── forge/
│   │       ├── core.hpp         # 核心功能接口
│   │       ├── math.hpp         # 数学工具函数
│   │       └── utils.hpp        # 通用工具
│   ├── core.cpp
│   ├── math.cpp
│   └── utils.cpp
├── app/                         # 可执行文件
│   ├── CMakeLists.txt
│   └── main.cpp
├── tests/                       # 单元测试
│   ├── CMakeLists.txt
│   ├── test_math.cpp
│   └── test_utils.cpp
└── .github/
    └── workflows/
        └── ci.yml               # GitHub Actions CI 配置
}

/// 技术要求

// CMake 配置要求

\list{
- `cmake_minimum_required(VERSION 3.25)`
- C++23 标准，`CMAKE_CXX_STANDARD_REQUIRED ON`，`CMAKE_CXX_EXTENSIONS OFF`
- `src/` 定义 STATIC 库目标 `forge`，头文件路径用 `PUBLIC` 暴露
- `app/` 定义可执行目标 `forge_app`，`PRIVATE` 链接 `forge` 库
- 严格编译警告（跨编译器兼容，用 generator expressions 区分 GCC/Clang 和 MSVC）
- Debug 模式自动启用 ASan + UBSan（仅 GCC/Clang）
- `FetchContent` 引入 fmt（日志输出）和 GoogleTest（单元测试）
- 启用 ccache（如果系统上可用）
- 预编译头（PCH）：常用标准库头文件
- `CMAKE_EXPORT_COMPILE_COMMANDS=ON`（为 clang-tidy 提供编译数据库）
}

// CMake Presets 要求

\table{
预设名称 | 构建类型 | 特殊配置
---
debug | Debug | `-O0 -g`，ASan + UBSan
release | Release | `-O2 -DNDEBUG`
asan | Debug（继承 debug） | 额外 `-fno-omit-frame-pointer`
tsan | Debug | ThreadSanitizer（替代 ASan）
profile | RelWithDebInfo | `-O2 -g`（保留符号用于 perf）
}

// 代码规范配置

\code[text]{
.clang-format:
  - BasedOnStyle: LLVM
  - IndentWidth: 4
  - ColumnLimit: 120
  - BreakBeforeBraces: Attach

.clang-tidy:
  - 启用 bugprone-*, modernize-*, performance-*, readability-*
  - 禁用 modernize-use-trailing-return-type
  - WarningsAsErrors: bugprone-*
  - HeaderFilterRegex 匹配项目头文件
}

// 代码内容要求

\list{
- `forge/math.hpp/cpp`：至少实现 3 个数学函数（如 `clamp`、`lerp`、`map_range`）
- `forge/utils.hpp/cpp`：至少实现 2 个工具函数（如 `to_upper`、`join_strings`）
- `forge/core.hpp/cpp`：定义一个 `Application` 类，演示 Pimpl 手法（头文件轻量化）
- `app/main.cpp`：使用 fmt 输出，调用库中的函数展示功能
- `tests/test_math.cpp`：至少 3 个 GoogleTest 测试用例
- `tests/test_utils.cpp`：至少 2 个 GoogleTest 测试用例
}

// CI 配置要求

\list{
- GitHub Actions：至少覆盖 Linux（GCC）和 Linux（Clang）两种编译器
- 构建流程：配置 → 编译（`-Werror`）→ 运行测试 → clang-format 检查
- 可选：Windows（MSVC）平台、ASan 专用任务、代码覆盖率
}

/// 实现提示

\steps{
\step[第一步：搭建项目骨架]{
创建目录结构，编写最小化的顶层 `CMakeLists.txt`——只包含 `project()`、标准设定和 `add_subdirectory()`。确保空项目能成功 `cmake --preset debug`。

\code[bash]{
mkdir -p ProjectForge/{src/include/forge,app,tests,.github/workflows}
cd ProjectForge
git init
}
}
\step[第二步：实现库和应用]{
实现 `forge` 库的源文件和头文件。先用简单函数验证 CMake 配置——能编译链接即可。

\code[cmake]{
# src/CMakeLists.txt 参考结构
add_library(forge STATIC core.cpp math.cpp utils.cpp)
target_include_directories(forge PUBLIC include)
target_compile_features(forge PUBLIC cxx_std_23)
target_precompile_headers(forge PRIVATE <vector> <string> <memory>)
}

注意 `core.hpp` 中使用 **Pimpl 手法**：头文件只暴露接口和一个 `unique_ptr<Impl>`，完整实现在 `core.cpp` 中。这样修改实现不需要重编依赖者。
}
\step[第三步：引入依赖与测试]{
用 `FetchContent` 拉取 fmt 和 GoogleTest。编写测试并确保 `ctest` 通过。

\code[cmake]{
# tests/CMakeLists.txt
add_executable(forge_tests test_math.cpp test_utils.cpp)
target_link_libraries(forge_tests PRIVATE forge GTest::gtest_main)
add_test(NAME ForgeTests COMMAND forge_tests)
}

\code[cpp]{
// test_math.cpp 示例
#include <gtest/gtest.h>
#include <forge/math.hpp>

TEST(MathTest, ClampInRange) {
    EXPECT_DOUBLE_EQ(forge::clamp(5.0, 0.0, 10.0), 5.0);
}
TEST(MathTest, ClampBelowMin) {
    EXPECT_DOUBLE_EQ(forge::clamp(-3.0, 0.0, 10.0), 0.0);
}
TEST(MathTest, LerpMidpoint) {
    EXPECT_DOUBLE_EQ(forge::lerp(0.0, 10.0, 0.5), 5.0);
}
}
}
\step[第四步：完善编译选项与预设]{
添加 generator expressions 处理跨编译器警告选项和 Sanitizer。编写完整的 `CMakePresets.json`，确保所有五个预设（debug / release / asan / tsan / profile）能正确构建。

\code[cmake]{
# 跨编译器警告选项
add_library(project_warnings INTERFACE)
target_compile_options(project_warnings INTERFACE
    $<$<CXX_COMPILER_ID:GNU,Clang>:-Wall -Wextra -Wpedantic -Wshadow -Wconversion>
    $<$<CXX_COMPILER_ID:MSVC>:/W4 /permissive->
)
}
}
\step[第五步：工具链集成]{
创建 `.clang-format` 和 `.clang-tidy` 配置文件。验证 `clang-format` 能格式化代码、`clang-tidy -p build/debug` 能运行检测。集成 ccache：

\code[cmake]{
find_program(CCACHE_FOUND ccache)
if(CCACHE_FOUND)
    set(CMAKE_CXX_COMPILER_LAUNCHER ccache)
endif()
}
}
\step[第六步：CI 配置]{
编写 `.github/workflows/ci.yml`，确保 push 时自动触发构建和测试。至少覆盖 Linux(GCC) 和 Linux(Clang)。
}
}

\warning{**常见陷阱：** 1) `FetchContent` 引入 GoogleTest 后，`GTest::gtest_main` 的目标名区分大小写。2) ASan 编译选项必须同时出现在 `target_compile_options` 和 `target_link_options` 中——只设一个会导致链接或运行时错误。3) ccache 检测用 `find_program`，不可写死路径。4) `#pragma once` 在所有头文件中使用，避免 include guard 命名冲突。}

\note{**Pimpl 实现要点：** `Application` 类的析构函数必须在 `core.cpp` 中定义（即使是 `= default`），因为析构 `unique_ptr<Impl>` 需要 `Impl` 的完整类型定义可见。如果在头文件中定义析构函数，编译器看不到 `Impl` 的完整定义会报错。}

/// 扩展挑战

\list{
- **三平台 CI**：GitHub Actions 完整覆盖 Linux + Windows + macOS
- **代码覆盖率**：集成 gcov + lcov（或 llvm-cov），生成覆盖率报告上传到 Codecov
- **Doxygen 文档**：为库函数添加 Doxygen 注释，CMake 中配置自动生成 HTML 文档
- **安装与导出**：`install(TARGETS ...)`、`install(EXPORT ...)` 使其他项目能 `find_package(ProjectForge)`
- **CPack 打包**：配置 CPack 生成 `.deb` / `.rpm` / `.zip` 发行包
- **动态库支持**：添加 `BUILD_SHARED_LIBS` 选项，同时支持静态和动态库构建
- **C++20 Modules**：将 `forge` 库的部分接口改用 Module 暴露（需 CMake 3.28+ 和 MSVC）
- **基准测试**：添加 `bench/` 目录，用 Google Benchmark 或 nanobench 测量数学函数性能
}
