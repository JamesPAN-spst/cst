/// 简介

上一节我们学习了 `new`/`delete` 的底层机制以及栈与堆的区别。你可能已经发现，默认的堆分配器（malloc/new）在每次分配时都要搜索空闲链表、可能触发系统调用，这在游戏引擎等每帧需要数千次分配的场景中远远不够快。

**分配器（Allocator）** 正是为了解决这个问题而生的——它让你接管内存分配的策略，用"预分配大块、内部自行管理"的方式获得极致的分配性能。本节从标准库分配器接口出发，带你理解线性分配器、栈分配器等经典策略，最终引入 C++17 的 PMR 框架，构建完整的分配器知识体系。

/// 知识点详解

// std::allocator 的基本接口

在理解自定义分配器之前，先看看标准库默认的 `std::allocator<T>` 做了什么。它是所有 STL 容器的默认**模板参数（Template Parameter）**，负责为容器分配和释放内存。

\code[cpp]{
// std::allocator 的核心接口（C++17 简化后）
#include <memory>

std::allocator<int> alloc;

// 分配 10 个 int 的原始内存（不调用构造函数）
int* p = alloc.allocate(10);

// 在原始内存上构造对象
std::allocator_traits<std::allocator<int>>::construct(alloc, p, 42);

// 析构对象（不释放内存）
std::allocator_traits<std::allocator<int>>::destroy(alloc, p);

// 释放原始内存
alloc.deallocate(p, 10);
}

`std::allocator` 本质上只是对 `::operator new` / `::operator delete` 的薄封装。它够用，但在高性能场景下有三个痛点：每次分配都走全局堆、无法控制内存来源、频繁小分配导致**内存碎片化（Memory Fragmentation）**。

// 为什么需要自定义分配器

\diagram[flow]{
默认分配路径: new/malloc -> 全局堆搜索空闲链表 -> 可能触发系统调用 -> 返回地址
自定义分配路径: allocate() -> 预分配缓冲区内移动指针 -> 返回地址
}

默认堆分配器在游戏引擎语境下有四个核心问题：

\table{
问题 | 描述 | 影响
---
速度不可预测 | malloc 需要搜索空闲链表，耗时不稳定 | 帧率抖动
碎片化 | 频繁分配/释放导致可用内存散落各处 | 长时间运行后分配失败
缓存不友好 | 分散的小对象破坏**空间局部性（Spatial Locality）** | cache miss 增加
线程竞争 | 全局堆需要加锁 | 多线程性能瓶颈
}

// 自定义分配器的编写方法

C++17 大幅简化了自定义分配器的接口。一个最小的分配器只需要提供 `allocate()`、`deallocate()` 和必要的类型别名：

\run[cpp]{
#include <iostream>
#include <vector>
#include <cstdlib>

template<typename T>
struct SimpleAllocator {
    using value_type = T;

    SimpleAllocator() = default;

    // 允许不同类型间转换（容器内部节点类型可能不同）
    template<typename U>
    SimpleAllocator(const SimpleAllocator<U>&) noexcept {}

    T* allocate(std::size_t n) {
        std::cout << "[SimpleAlloc] allocate " << n << " x " << sizeof(T)
                  << " = " << n * sizeof(T) << " bytes\n";
        return static_cast<T*>(std::malloc(n * sizeof(T)));
    }

    void deallocate(T* p, std::size_t n) noexcept {
        std::cout << "[SimpleAlloc] deallocate " << n * sizeof(T) << " bytes\n";
        std::free(p);
    }
};

template<typename T, typename U>
bool operator==(const SimpleAllocator<T>&, const SimpleAllocator<U>&) { return true; }

int main() {
    std::vector<int, SimpleAllocator<int>> v;
    v.push_back(10);
    v.push_back(20);
    v.push_back(30);
    std::cout << "size = " << v.size() << "\n";
    return 0;
}
}

运行后可以看到 `vector` 扩容时的分配模式——每次容量不够，就分配更大的块并释放旧块。

\warning{**跨类型转换构造函数不可省略。** STL 容器在内部可能使用与 `T` 不同的类型（例如 `std::list<int>` 内部分配的是链表节点而非 `int`），分配器必须支持从 `SimpleAllocator<U>` 到 `SimpleAllocator<T>` 的转换。}

// 线性分配器（Linear/Bump Allocator）

**线性分配器（Linear Allocator）**，又称 **Bump Allocator**，是所有自定义分配器中最简单、最快的。核心思想：维护一个指针，每次分配只向前移动指针，不支持单个释放，只能整体重置。

\diagram[memory]{
buffer (预分配 1MB):
  [0x0000] 已分配区域 A (64B)
  [0x0040] 已分配区域 B (128B)
  [0x00C0] 已分配区域 C (32B)
  [0x00E0] ← offset 指针（下次分配从这里开始）
  [0x00E0 ... 0xFFFFF] 空闲区域
arrow: offset -> 下次分配位置
}

\run[cpp]{
#include <iostream>
#include <cstdlib>
#include <cstdint>

class LinearAllocator {
    char* buffer_;
    std::size_t capacity_;
    std::size_t offset_ = 0;

public:
    explicit LinearAllocator(std::size_t size)
        : buffer_(static_cast<char*>(std::malloc(size)))
        , capacity_(size) {}

    ~LinearAllocator() { std::free(buffer_); }

    // 分配：对齐 + 移动指针，O(1)
    void* allocate(std::size_t size,
                   std::size_t alignment = alignof(std::max_align_t)) {
        // 将 offset 向上对齐到 alignment 的倍数
        std::size_t aligned = (offset_ + alignment - 1) & ~(alignment - 1);
        if (aligned + size > capacity_) return nullptr;
        void* ptr = buffer_ + aligned;
        offset_ = aligned + size;
        return ptr;
    }

    void reset() { offset_ = 0; }
    std::size_t used() const { return offset_; }
    std::size_t remaining() const { return capacity_ - offset_; }
};

int main() {
    LinearAllocator alloc(1024);

    int* a = static_cast<int*>(alloc.allocate(sizeof(int) * 4, alignof(int)));
    double* b = static_cast<double*>(alloc.allocate(sizeof(double) * 2, alignof(double)));

    std::cout << "used: " << alloc.used() << " / 1024 bytes\n";
    std::cout << "remaining: " << alloc.remaining() << " bytes\n";

    alloc.reset();
    std::cout << "after reset, used: " << alloc.used() << " bytes\n";
    return 0;
}
}

\tip{**线性分配器是游戏引擎中的"帧分配器（Frame Allocator）"。** 每帧开始时分配临时数据（渲染命令、调试字符串等），帧结束时 `reset()` 一键释放。分配只需移动指针——比 malloc 快一个数量级。}

\danger{**线性分配器中的非平凡对象必须手动析构。** `reset()` 只是把 offset 归零，不会调用任何析构函数。如果你用 placement new 构造了 `std::string` 或其他持有资源的对象，必须在 reset 前逐个调用析构函数，否则资源泄漏。}

// 栈分配器（Stack Allocator）

**栈分配器（Stack Allocator）** 在线性分配器基础上增加了一个能力：按**后进先出（LIFO）**的顺序释放。它通过 **标记（Marker）** 机制记录分配位置，支持回退到任意标记点。

\code[cpp]{
class StackAllocator {
    char* buffer_;
    std::size_t capacity_;
    std::size_t offset_ = 0;

public:
    using Marker = std::size_t;

    explicit StackAllocator(std::size_t size)
        : buffer_(static_cast<char*>(std::malloc(size)))
        , capacity_(size) {}

    ~StackAllocator() { std::free(buffer_); }

    Marker getMarker() const { return offset_; }

    void* allocate(std::size_t size,
                   std::size_t alignment = alignof(std::max_align_t)) {
        std::size_t aligned = (offset_ + alignment - 1) & ~(alignment - 1);
        if (aligned + size > capacity_) return nullptr;
        void* ptr = buffer_ + aligned;
        offset_ = aligned + size;
        return ptr;
    }

    // 回退到标记点——释放标记之后的所有分配
    void freeToMarker(Marker marker) { offset_ = marker; }
    void reset() { offset_ = 0; }
    std::size_t used() const { return offset_; }
};
}

典型用法是嵌套作用域——例如物理引擎碰撞检测分粗筛、细筛阶段，每个阶段的临时数据在阶段结束后释放：

\code[cpp]{
StackAllocator stack(65536);

// 粗筛阶段
auto mark1 = stack.getMarker();
auto* broadResults = static_cast<AABB*>(
    stack.allocate(maxPairs * sizeof(AABB)));
// ... 粗筛计算 ...

    // 细筛阶段（嵌套）
    auto mark2 = stack.getMarker();
    auto* contacts = static_cast<Contact*>(
        stack.allocate(narrowCount * sizeof(Contact)));
    // ... 细筛计算 ...
    stack.freeToMarker(mark2); // 释放细筛临时数据

stack.freeToMarker(mark1); // 释放粗筛临时数据
}

\note{**栈分配器 vs 线性分配器：** 线性分配器只能整体 reset，栈分配器可以按 LIFO 顺序逐层释放。代价是需要维护 Marker。如果你的场景只需要"帧级别统一释放"，线性分配器更简单；如果需要嵌套作用域释放，选栈分配器。}

// 分配器与 STL 容器

自定义分配器最大的价值之一是与 STL 容器配合使用。但传统模型有一个严重问题：

\code[cpp]{
// 传统方式：分配器作为模板参数
// 不同分配器的 vector 是不同类型！
std::vector<int, MyAllocA<int>> va; // 类型 A
std::vector<int, MyAllocB<int>> vb; // 类型 B
// va = vb; // 编译错误！类型不兼容

// 函数签名也受污染：
// void process(std::vector<int, ???>& v);
// 只能模板化或固定分配器类型
}

这个"分配器类型污染容器类型"的问题是 C++98/11 分配器模型的最大痛点，也是 C++17 引入 PMR 的核心动机。

// PMR（Polymorphic Memory Resources，C++17）

**PMR（多态内存资源）** 通过**运行时多态（Runtime Polymorphism）**而非编译期模板参数来指定分配策略，彻底解决了类型污染问题。

\compare{
old: C++98/11 的分配器是模板参数，不同分配器 → 不同容器类型。接口极复杂（需十几个 typedef）。实践中几乎没人自定义分配器。

new: C++17 的 `std::pmr` 用 `memory_resource*` 指针传递分配策略，不影响容器类型。自定义分配器只需继承 `std::pmr::memory_resource` 并重写 3 个虚函数。
}

PMR 的核心是 `std::pmr::memory_resource` 抽象基类：

\code[cpp]{
// <memory_resource> 头文件
class memory_resource {
public:
    void* allocate(size_t bytes, size_t alignment);
    void deallocate(void* p, size_t bytes, size_t alignment);
    bool is_equal(const memory_resource& other) const noexcept;

protected:
    // 自定义时只需重写这三个虚函数
    virtual void* do_allocate(size_t bytes, size_t alignment) = 0;
    virtual void do_deallocate(void* p, size_t bytes, size_t alignment) = 0;
    virtual bool do_is_equal(const memory_resource& other) const noexcept = 0;
};
}

// std::pmr::monotonic_buffer_resource 详解

`monotonic_buffer_resource` 本质上就是标准库提供的**线性分配器**——只分配不释放，直到 `release()` 一键重置。

\run[cpp]{
#include <iostream>
#include <memory_resource>
#include <vector>
#include <string>

int main() {
    // 在栈上准备一块缓冲区
    char buffer[4096];

    // monotonic_buffer_resource：从 buffer 分配
    // 溢出时回退到 new_delete_resource（堆）
    std::pmr::monotonic_buffer_resource pool(
        buffer, sizeof(buffer),
        std::pmr::new_delete_resource()
    );

    // pmr::vector 通过指针接受分配策略
    std::pmr::vector<int> nums(&pool);
    nums.reserve(100);
    for (int i = 0; i < 100; ++i) {
        nums.push_back(i);
    }

    std::pmr::string msg("Hello from pmr!", &pool);

    std::cout << "vector size: " << nums.size() << "\n";
    std::cout << "string: " << msg << "\n";
    return 0;
}
}

\tip{**用 `null_memory_resource()` 做溢出保护。** 将上游设为 `std::pmr::null_memory_resource()`，溢出时抛出 `std::bad_alloc` 而非悄悄回退到堆。这在调试阶段帮你发现缓冲区预估不足的问题。}

// 自定义 memory_resource 实战

下面实现一个带日志追踪的分配器，演示如何继承 `memory_resource`：

\run[cpp]{
#include <iostream>
#include <memory_resource>
#include <vector>

class LoggingResource : public std::pmr::memory_resource {
    std::pmr::memory_resource* upstream_;
    std::size_t totalBytes_ = 0;
    int allocCount_ = 0;

protected:
    void* do_allocate(std::size_t bytes, std::size_t alignment) override {
        totalBytes_ += bytes;
        ++allocCount_;
        std::cout << "[alloc #" << allocCount_ << "] "
                  << bytes << " bytes (align " << alignment << ")\n";
        return upstream_->allocate(bytes, alignment);
    }

    void do_deallocate(void* p, std::size_t bytes,
                       std::size_t alignment) override {
        upstream_->deallocate(p, bytes, alignment);
    }

    bool do_is_equal(const std::pmr::memory_resource& other)
        const noexcept override {
        return this == &other;
    }

public:
    explicit LoggingResource(
        std::pmr::memory_resource* up = std::pmr::get_default_resource())
        : upstream_(up) {}

    void report() const {
        std::cout << "=== Total: " << totalBytes_
                  << " bytes in " << allocCount_ << " calls ===\n";
    }
};

int main() {
    LoggingResource logger;
    {
        std::pmr::vector<int> v(&logger);
        for (int i = 0; i < 20; ++i) v.push_back(i);
    }
    logger.report();
    return 0;
}
}

PMR 的强大之处在于**分配器可以链式组合**——例如 `LoggingResource → monotonic_buffer_resource → new_delete_resource`，日志层追踪、monotonic 层快速分配、堆层兜底。

// 游戏引擎中的分配器选型策略

\table{
分配器类型 | 分配速度 | 释放方式 | 碎片 | 典型用途
---
线性/帧 | O(1) 极快 | 整体 reset | 零 | 帧临时数据、渲染命令
栈 | O(1) | LIFO 标记回退 | 零 | 嵌套阶段临时数据
空闲链表池 | O(1) | O(1) 单个释放 | 零（固定大小） | 粒子、子弹、Entity
`pmr::monotonic` | O(1) | release 全部释放 | 零 | 标准库容器的帧分配
`pmr::pool` | 近 O(1) | 按大小分池释放 | 低 | 多种大小的小对象
}

\diagram[flow]{
游戏引擎分配器决策:
  需要分配内存 -> 临时数据(仅本帧)? -是-> 线性分配器
  临时数据(仅本帧)? -否-> 固定大小对象? -是-> 对象池
  固定大小对象? -否-> 需要LIFO释放? -是-> 栈分配器
  需要LIFO释放? -否-> 通用堆(malloc)
}

\collapse[深入：分配器链与上游策略]{
在真实游戏引擎中，分配器不直接调用 `malloc`，而是从预分配的**内存区域（Memory Arena）**获取底层内存，形成分层体系：

1. **系统层：** 进程启动时通过 `mmap`/`VirtualAlloc` 一次性申请大块内存（64MB~512MB）
2. **Arena 层：** 将大块内存划分给不同子系统（渲染 16MB、物理 8MB、音频 4MB）
3. **分配器层：** 每个子系统在自己的 Arena 内使用线性/池/栈分配器
4. **应用层：** 游戏逻辑通过分配器获取内存

好处：每个子系统的内存用量可追踪、可限制，不会互相干扰。也便于 profiling 时分析内存热点。
}

/// 工程实践

\list{
- **优先使用 `std::pmr` 而非传统分配器模板参数。** PMR 不污染容器类型，API 设计更简洁。
- **帧分配器用 `monotonic_buffer_resource` + `null_memory_resource` 上游。** 溢出直接报错，帮你发现缓冲区不足。
- **Release 模式下关闭日志分配器。** 日志层只在 Debug/Profile 构建中启用。
- **开发阶段用 `LoggingResource` 统计每帧实际分配量。** 然后为生产构建固定一个略有余量的缓冲区大小。
- **分配器生命周期必须长于使用它的所有容器。** `memory_resource*` 是裸指针，没有所有权语义。
}

/// 难点易错点分析

\danger{**pmr 容器与普通容器是不同类型！** `std::vector<int>` 和 `std::pmr::vector<int>` 不能隐式转换。设计接受容器的函数时，要么用模板泛化，要么统一使用 pmr 版本。}

\danger{**monotonic_buffer_resource 的 deallocate() 是空操作！** 它只在 `release()` 时释放所有内存。容器频繁扩容时，旧内存不会回收（直到 release），内存会持续增长。这是设计意图而非 bug。}

\warning{**自定义 memory_resource 的 do_is_equal() 不能随意返回 true。** 两个不同的 `memory_resource` 声明相等意味着用 A 分配的内存可以用 B 释放。只有真正无状态、可互换的分配器才应返回 true。}

\quiz[choice]{
Q: 关于 `std::pmr::monotonic_buffer_resource`，哪一项是**正确**的？
A: 每次调用 deallocate() 会立即归还内存
B: 只有调用 release() 才会释放所有已分配的内存
C: 它是线程安全的，可在多线程中直接使用
D: 它不支持设置上游 memory_resource
ANS: B
EXP: monotonic_buffer_resource 的 deallocate() 是空操作，内存只在 release()（或析构）时统一释放。它非线程安全（需自行加锁或用 synchronized_pool_resource）。它支持上游资源作为溢出回退。
}

\quiz[tf]{
Q: 使用 C++17 PMR 后，`std::pmr::vector<int>` 即使使用不同的 `memory_resource`，其**静态类型**仍然相同。
ANS: T
EXP: 这是 PMR 的核心优势。分配策略通过运行时指针传递，不影响容器的静态类型。`std::pmr::vector<int>` 就是 `std::vector<int, std::pmr::polymorphic_allocator<int>>`，分配器类型始终固定。
}

\quiz[choice]{
Q: 游戏引擎中，最适合存储"仅本帧有效的临时渲染命令"的分配器是？
A: std::allocator（默认堆分配）
B: 线性分配器 / 帧分配器
C: 伙伴系统（Buddy System）
D: 空闲链表池（Free List Pool）
ANS: B
EXP: 帧临时数据的特点是"批量分配、帧末统一释放"，线性分配器完美匹配——分配 O(1)，释放只需 reset()，零碎片。空闲链表池适合固定大小的长生命周期对象。
}

// 🟢 基础题：给线性分配器添加使用统计

\run[cpp]{
#include <iostream>
#include <cstdlib>

class LinearAllocator {
    char* buffer_;
    std::size_t capacity_;
    std::size_t offset_ = 0;
    int allocCount_ = 0;
    std::size_t peakUsage_ = 0;

public:
    explicit LinearAllocator(std::size_t size)
        : buffer_(static_cast<char*>(std::malloc(size)))
        , capacity_(size) {}

    ~LinearAllocator() { std::free(buffer_); }

    void* allocate(std::size_t size,
                   std::size_t alignment = alignof(std::max_align_t)) {
        std::size_t aligned = (offset_ + alignment - 1) & ~(alignment - 1);
        if (aligned + size > capacity_) return nullptr;
        void* ptr = buffer_ + aligned;
        offset_ = aligned + size;
        ++allocCount_;
        if (offset_ > peakUsage_) peakUsage_ = offset_;
        return ptr;
    }

    void reset() { offset_ = 0; }

// 🔴 挑战题：分配器性能对比平台

设计基准测试程序，对比三种策略在"模拟游戏帧"场景下的性能：
1. 默认 `std::allocator`
2. `std::pmr::monotonic_buffer_resource`
3. 你自己实现的 `LinearAllocator`

每帧操作：创建 vector 存 1000 个 int + 100 个短字符串。运行 10000 帧，用 `<chrono>` 测量总耗时，输出格式化对比报告。

\collapse[参考思路]{
- 使用 `std::chrono::high_resolution_clock` 计时
- monotonic 版本每帧前 `release()`，LinearAllocator 版本每帧前 `reset()`
- 默认分配器版本正常创建销毁局部容器
- 编译时用 `-O2` 优化，否则结果无参考性
- PMR 版本使用 `pmr::vector` 和 `pmr::string`
- LinearAllocator 版本需手动 placement new
}
