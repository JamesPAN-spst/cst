/// 简介

上一节我们学了 `constexpr` 让计算在编译期完成，这一节来解决模板编程中另一个长期痛点——**类型约束**。

想象一下：你写了一个模板函数 `template<typename T> T lerp(T a, T b, T t)`，本意是给数值类型做线性插值（Linear Interpolation）。但有人传进来一个 `std::string`，编译器吐出 200 行看不懂的错误。在 C++20 之前，限制模板参数类型只能靠 **SFINAE**（Substitution Failure Is Not An Error，替换失败不是错误）——那是一种"故意让编译器在特定情况下匹配失败"的黑魔法，代码丑陋且错误信息更难读。

**概念（Concepts）** 是 C++20 引入的正式机制：用一行清晰的声明告诉编译器"T 必须满足什么条件"。如果不满足，编译器直接说"T 不满足 Numeric 约束"，而不是把模板实例化的整棵推导树全倒出来。

本节从 SFINAE 的痛点出发，逐步学习 `requires` 表达式、`concept` 定义、标准库常用概念、三种约束语法、约束组合、包含（Subsumption）规则，以及 concepts 与 `auto` 的配合。

/// 知识点详解

// 为什么需要 Concepts

在游戏开发中，模板函数几乎无处不在——向量运算、资源管理、事件系统。但 C++17 之前，"限制模板参数类型"只有 SFINAE 一条路。来看一个典型的痛点：

\tabs{
\tab[C++17 SFINAE]{
\code[cpp]{
#include <type_traits>

// 想限制 T 必须是数值类型，只能这样写：
template<typename T,
         typename = std::enable_if_t<std::is_arithmetic_v<T>>>
T lerp(T a, T b, T t) {
    return a + t * (b - a);
}
// 传入 std::string 时，错误信息可能长达几十行
// 且完全不会提到 "T 不是算术类型"
}
}
\tab[C++20 Concepts]{
\code[cpp]{
#include <concepts>

// 一眼看出：T 必须是整数或浮点
template<typename T>
requires std::integral<T> || std::floating_point<T>
T lerp(T a, T b, T t) {
    return a + t * (b - a);
}
// 传入 std::string 时，编译器直说：
// "constraints not satisfied: T = std::string"
}
}
}

上面的对比展示了 Concepts 最直接的好处：**代码更好写、错误更好读**。

// requires 表达式

**requires 表达式（requires-expression）** 是 Concepts 的基础构件。它描述"对类型 T 的一组要求"，编译器检测这些要求是否成立，返回 `true` 或 `false`。

requires 表达式有四种要求形式：

\code[cpp]{
template<typename T>
concept GameComponent = requires(T comp, float dt) {
    // 1. 简单要求（Simple Requirement）：表达式能通过编译即可
    comp.update(dt);
    comp.render();

    // 2. 类型要求（Type Requirement）：嵌套类型必须存在
    typename T::ComponentID;

    // 3. 复合要求（Compound Requirement）：表达式合法 + 返回类型约束
    { comp.name() } -> std::convertible_to<std::string_view>;
    { comp.priority() } -> std::integral;

    // 4. 嵌套要求（Nested Requirement）：里面再写 requires 子句
    requires std::movable<T>;
    requires sizeof(T) <= 256;  // 组件不要太大
};
}

逐条解释：
- **简单要求**只检查"表达式能否编译"，不关心返回类型。
- **类型要求**用 `typename T::某类型` 检查嵌套类型是否存在。
- **复合要求**用 `{ expr } -> concept<...>` 同时检查表达式合法和返回类型。箭头 `->` 后面跟的是一个对返回类型的约束。
- **嵌套要求**用 `requires` 关键字嵌入额外约束，可以是已有的 concept 或编译期布尔表达式。

\run[cpp]{
#include <concepts>
#include <iostream>
#include <string>
#include <vector>

// 定义一个简单的 requires 表达式来测试
template<typename T>
concept HasSize = requires(T t) {
    { t.size() } -> std::convertible_to<std::size_t>;
};

// 用概念约束函数
void printSize(HasSize auto const& container) {
    std::cout << "size = " << container.size() << "\n";
}

int main() {
    std::vector<int> v{1, 2, 3};
    std::string s = "hello";

    printSize(v);  // OK: vector 有 size()
    printSize(s);  // OK: string 有 size()
    // printSize(42);  // 取消注释试试：int 没有 size()
    std::cout << "All checks passed!\n";
}
}

// concept 定义语法

定义一个**概念（concept）**的语法很直接：

\code[cpp]{
template<typename T>
concept 概念名 = 约束表达式;
}

约束表达式可以是：一个 `requires` 表达式、已有的 concept、编译期布尔值，或者它们的组合。

\code[cpp]{
// 基于已有 concept 组合
template<typename T>
concept Numeric = std::integral<T> || std::floating_point<T>;

// 基于 requires 表达式
template<typename T>
concept Serializable = requires(T obj, std::ostream& os) {
    { os << obj } -> std::same_as<std::ostream&>;
};

// 基于编译期布尔
template<typename T>
concept SmallType = sizeof(T) <= 16;

// 混合组合
template<typename T>
concept FastComponent = Numeric<T> && SmallType<T>;
}

concept 本身是一个编译期布尔常量——你可以在 `static_assert` 或 `if constexpr` 中直接使用它：

\code[cpp]{
static_assert(Numeric<int>);          // 通过
static_assert(!Numeric<std::string>); // 通过
}

// 标准库常用 concepts

C++20 在 `<concepts>` 头文件中提供了大量现成的概念，不需要自己重新发明：

\table{
头文件 | concept | 含义
---
`<concepts>` | `std::integral<T>` | T 是整型（int, long, char 等）
`<concepts>` | `std::floating_point<T>` | T 是浮点型（float, double）
`<concepts>` | `std::same_as<T, U>` | T 和 U 是完全相同的类型
`<concepts>` | `std::derived_from<D, B>` | D 继承自 B
`<concepts>` | `std::convertible_to<From, To>` | From 可隐式转换为 To
`<concepts>` | `std::movable<T>` | T 可移动
`<concepts>` | `std::copyable<T>` | T 可拷贝
`<concepts>` | `std::regular<T>` | 可默认构造 + 可拷贝 + 可比较相等
`<concepts>` | `std::invocable<F, Args...>` | F 可用 Args... 调用
`<concepts>` | `std::predicate<F, Args...>` | F(Args...) 返回 bool
`<iterator>` | `std::input_iterator<I>` | 输入迭代器
`<iterator>` | `std::random_access_iterator<I>` | 随机访问迭代器
`<ranges>` | `std::ranges::range<R>` | R 是一个范围（有 begin/end）
`<ranges>` | `std::ranges::sized_range<R>` | R 是有 size() 的范围
}

\tip{游戏开发中最常用的：`std::integral`（整数 ID）、`std::floating_point`（坐标/物理值）、`std::invocable`（回调函数/事件处理）、`std::ranges::range`（批量处理实体）。}

// 约束函数模板——三种语法

约束函数模板有三种等效的写法，选哪种看场景和个人偏好：

\run[cpp]{
#include <concepts>
#include <iostream>

template<typename T>
concept Numeric = std::integral<T> || std::floating_point<T>;

// 语法一：requires 子句（Requires Clause）—— 最明确
template<typename T>
requires Numeric<T>
T add(T a, T b) { return a + b; }

// 语法二：尾部约束（Trailing Requires）—— 约束写在参数列表之后
template<typename T>
T multiply(T a, T b) requires Numeric<T> { return a * b; }

// 语法三：缩写模板（Abbreviated Template）—— 最简洁
auto square(Numeric auto x) { return x * x; }

int main() {
    std::cout << "add:      " << add(3, 4) << "\n";
    std::cout << "multiply: " << multiply(2.5, 3.0) << "\n";
    std::cout << "square:   " << square(7) << "\n";
}
}

三种语法的选用建议：
- **requires 子句**：当约束复杂或需要组合多个条件时最清晰。
- **尾部约束**：当模板参数列表已经很长，约束放后面更易读。
- **缩写模板**：对简单的单参数模板最方便，一行搞定。

\warning{缩写模板中，每个 `Numeric auto` 参数是**独立的模板参数**。`auto square(Numeric auto a, Numeric auto b)` 中 `a` 和 `b` 可以是不同类型（如 `int` 和 `double`）。如果要求它们必须相同类型，请用 `template<Numeric T> auto square(T a, T b)`。}

// 约束类模板

concept 不仅能约束函数，也能约束类模板参数：

\code[cpp]{
#include <concepts>
#include <vector>

template<typename T>
concept Component = std::movable<T> && std::default_initializable<T>;

// 只有满足 Component 约束的类型才能实例化 ComponentPool
template<Component C>
class ComponentPool {
    std::vector<C> pool_;
public:
    void add(C comp) { pool_.push_back(std::move(comp)); }
    C& get(std::size_t idx) { return pool_[idx]; }
    std::size_t count() const { return pool_.size(); }
};
}

还可以对类模板的**个别成员函数**施加额外约束：

\code[cpp]{
template<Component C>
class ComponentPool {
    std::vector<C> pool_;
public:
    void add(C comp) { pool_.push_back(std::move(comp)); }

    // 只有当 C 还满足 equality_comparable 时，才存在 find 方法
    std::size_t find(const C& target) const
        requires std::equality_comparable<C>
    {
        for (std::size_t i = 0; i < pool_.size(); ++i)
            if (pool_[i] == target) return i;
        return pool_.size();  // not found
    }
};
}

这样，如果组件类型不支持 `==`，你仍然可以用 `add` 和 `get`，只是调不了 `find` 而已。

// concept 组合（&& 和 ||）

concept 支持逻辑与（`&&`）和逻辑或（`||`）组合，让你从小概念搭出大概念：

\code[cpp]{
template<typename T>
concept Arithmetic = std::integral<T> || std::floating_point<T>;

template<typename T>
concept Vec3Like = requires(T v) {
    { v.x } -> std::convertible_to<float>;
    { v.y } -> std::convertible_to<float>;
    { v.z } -> std::convertible_to<float>;
};

// 游戏中的可渲染组件：必须有位置，且有 render 方法
template<typename T>
concept Renderable = Vec3Like<T> && requires(T obj) {
    { obj.render() } -> std::same_as<void>;
};
}

\note{concept 组合是 Concepts 最强大的设计之一——你可以建立一套自定义概念体系（类似接口层次），但它是编译期零开销的，不需要虚函数。}

// 包含规则（Subsumption）与重载决议

当两个重载函数的约束有"包含"关系时，编译器选择**更严格的那个**。这叫做**包含规则（Subsumption）**：

\run[cpp]{
#include <concepts>
#include <iostream>

template<typename T>
concept Arithmetic = std::integral<T> || std::floating_point<T>;

template<typename T>
concept SignedNum = Arithmetic<T> && std::signed_integral<T>;
// SignedNum 包含了 Arithmetic 的所有约束，还额外要求 signed

// 两个重载
void process(Arithmetic auto x) {
    std::cout << "Arithmetic: " << x << "\n";
}
void process(SignedNum auto x) {
    std::cout << "SignedNum:  " << x << "\n";
}

int main() {
    process(42);    // int 同时满足两者 -> 选更严格的 SignedNum
    process(42u);   // unsigned 只满足 Arithmetic
    process(3.14);  // double 满足 Arithmetic 但不满足 signed_integral
}
}

包含规则只在约束通过**逻辑组合**（`&&`、`||`）建立从属关系时生效。如果两个 concept 碰巧约束了相同条件但定义上没有引用关系，编译器不认为它们有包含关系，会报歧义错误（ambiguity）。

\quiz[choice|单选]{
Q: 给定 `concept A = std::integral<T>;` 和 `concept B = std::integral<T> && std::signed_integral<T>;`，调用 `f(42)` 时 `void f(A auto)` 和 `void f(B auto)` 哪个被选中？
A: f(A auto)，因为 A 先声明
B: f(B auto)，因为 B 更严格（subsumption）
C: 编译错误：歧义
D: 未定义行为
ANS: B
EXP: B 的约束包含了 A 的全部约束并额外加上 `signed_integral`，因此 B 比 A 更严格。根据包含规则，编译器选择约束更强的重载。
}

// concepts 与 auto

C++20 允许在 `auto` 前面加上 concept 名来**约束自动推导的类型**，不仅限于函数参数：

\code[cpp]{
// 函数参数（缩写模板）
void draw(std::floating_point auto x, std::floating_point auto y);

// 变量声明
std::integral auto count = getEntityCount();  // 确保返回的是整型

// 返回类型
std::floating_point auto computeDelta(float dt) {
    return dt * 0.016f;
}

// lambda 参数
auto clamp = [](std::totally_ordered auto val,
                std::totally_ordered auto lo,
                std::totally_ordered auto hi) {
    return val < lo ? lo : (val > hi ? hi : val);
};
}

\quiz[tf|判断]{
Q: `std::integral auto x = 3.14;` 能通过编译。
ANS: F
EXP: `3.14` 被推导为 `double`，不满足 `std::integral` 约束，编译失败。这正是 concepts + auto 的价值所在——在变量声明时就能捕获类型错误。
}

/// 新版特性（C++17/20/23）

\compare{
old: C++17 及之前，约束模板参数只有 SFINAE / `enable_if` / `static_assert` 三板斧。SFINAE 代码晦涩难读，`enable_if` 嵌套在模板参数列表中极不直观，`static_assert` 虽然消息友好但无法用于重载决议。一个简单的"T 必须是数值类型"可能需要三四行辅助代码。

new: C++20 Concepts 一步到位解决了可读性、错误信息、重载决议三个问题。`requires` 表达式和 `concept` 关键字是标准语法，不再是"技巧"。C++23 进一步收紧了 subsumption 规则中的一些边界情况，并且标准库添加了更多预定义概念（如 `std::formattable`）。
}

\collapse[C++23 新增概念一览]{
C++23 新增了一些实用概念：
- `std::formattable<T, CharT>`：T 可以被 `std::format` 格式化
- 对 ranges 体系的概念做了补充和完善
- 澄清了 subsumption 在折叠表达式中的行为

这些属于渐进式改进，核心语法仍然是 C++20 定义的那一套。
}

/// 工程实践

\tip{
**命名约定**：自定义 concept 建议使用形容词或"能力名词"命名，首字母大写。例如 `Renderable`、`Serializable`、`HasPosition`。避免用 `ConceptXxx` 前缀——concept 本身就该像形容词一样修饰类型。
}

\tip{
**小概念组合优于大概念**：不要写一个巨大的 `GameEntity` concept 把所有要求塞进去。拆成 `Movable`、`Renderable`、`Collidable` 等小概念，用 `&&` 按需组合。这样每个概念可以独立复用，测试也更方便。
}

\note{
**何时用 concept vs 基类接口（虚函数）？**
- 如果对象类型在**编译期确定**——用 concept（零开销，更好的优化）
- 如果对象类型在**运行时才知道**（如从配置文件加载插件）——用虚函数
- 游戏引擎中 ECS 的组件类型通常编译期已知，非常适合 concept
}

/// 难点易错点分析

\danger{
**`requires requires` 双关键字陷阱：** 当你在 requires 子句中直接写 requires 表达式，会出现 `requires requires`——第一个是"子句关键字"，第二个是"表达式关键字"：

\code[cpp]{
template<typename T>
void f(T x) requires requires(T a, T b) { a + b; };
//              ^子句       ^表达式
}

这在语法上完全合法，但可读性极差。**最佳实践：把 requires 表达式提取为具名 concept**，避免 `requires requires`。
}

\danger{
**concept 只检查语法，不检查语义！** `requires(T a) { a + a; }` 只验证 `+` 运算符存在，不验证 `+` 的行为是否符合数学加法。一个类型可能定义了 `operator+` 但做了完全不相关的事情。语义正确性必须靠文档和测试保证。
}

\danger{
**约束不参与隐式转换：** `std::integral<T>` 要求 T 本身是整型，不接受"可转到整型"的类型。即使你的类有 `operator int()`，也不会满足 `std::integral`。如果需要允许转换，要写 `std::convertible_to<T, int>`。
}

\quiz[choice|单选]{
Q: 以下代码能否编译？
`template<typename T> concept Addable = requires(T a) { a + a; };`
`static_assert(Addable<std::string>);`
A: 能编译且 static_assert 通过，因为 string 支持 operator+
B: 能编译但 static_assert 失败，因为 string 不是数值类型
C: 编译错误，concept 不能用在 static_assert 中
D: 未定义行为
ANS: A
EXP: `Addable` 只要求 `a + a` 表达式合法。`std::string` 重载了 `operator+`（字符串拼接），所以满足约束。这正是"concept 只检语法不检语义"的体现。
}

\quiz[tf|判断]{
Q: 缩写模板 `void f(std::integral auto a, std::integral auto b)` 保证 `a` 和 `b` 是相同类型。
ANS: F
EXP: 每个 `auto` 是独立的模板参数。`a` 可能是 `int` 而 `b` 可能是 `long`。要保证相同类型，必须写成 `template<std::integral T> void f(T a, T b)`。
}

\run[cpp]{
#include <concepts>
#include <iostream>

// safeDiv 只接受整型参数
// safeDivF 只接受浮点型参数

    if (b == 0) return 0;
    return a / b;

    return a / b;

    std::cout << safeDivF(10.0, 3.0) << "\n";  // 应输出 3.33...
    std::cout << "Done!\n";
}

// 🔴 挑战题：类型安全的事件系统

设计一个基于 Concepts 的**编译期类型安全事件系统**，满足以下需求：

1. 定义 `Event` concept：要求有 `static constexpr int event_id` 成员和 `name()` 方法
2. 定义 `EventHandler` concept：要求是一个可以用 `const Event auto&` 调用的可调用对象
3. 实现 `EventBus` 类模板，用 concept 约束注册（subscribe）和触发（publish）操作
4. 至少支持两种事件类型（如 `CollisionEvent` 和 `DamageEvent`）
5. 如果用户注册了错误类型的 handler，编译期就要报错

验收标准：
- 编译通过（C++20），演示注册处理器和触发事件
- 尝试注册不匹配的 handler 时，编译器给出清晰的约束不满足错误
- 利用 concept 组合而非 SFINAE

\collapse[参考思路]{
核心架构：
1. `Event` concept 检查 `T::event_id` 和 `t.name()` 是否合法
2. `EventHandler<E>` 用 `std::invocable<F, const E&>` 约束
3. `EventBus` 内部可用 `std::function` + `std::unordered_map<int, ...>` 存储 handler
4. `subscribe<E>(handler)` 和 `publish(const Event auto& e)` 都用 concept 约束
5. 可以用 `if constexpr` + concept 做更精细的分发

关键点：concept 保证类型安全发生在**编译期**，运行时没有类型检查开销。
}
