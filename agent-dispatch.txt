本文件说明：
  1. 各 Agent 的单元分派情况（共 84 个单元，84 位 Agent）。
  2. 每位 Agent 的完整工作流程（输入 → 执行 → 输出）。
  3. 用户如何查看、管理和验收 Agent 的产出。

================================================================================
  一、Agent 工作总则
================================================================================

每位 Agent 只负责一个单元（一个 .txt 文件），独立工作，互不干扰。

■ 每位 Agent 的三个必读文件：
  1. rule.txt            — 写作规范、标记语法、评分标准（必须严格遵守）
  2. 对应模块的现有 .txt — 作为起点，保留已有高质量内容，不得无故删改
  3. report-template.txt — 工作完成后按此模板填写报告

■ 产出要求（每位 Agent 必须提交以下两项）：
  A. 更新后的单元 .txt 文件（位于 content/XX-module/ 目录下）
  B. 一份按 report-template.txt 填写的报告（写入 report.txt 对应条目，
     得分记录到 workflow.txt）

■ 质量门槛：
  所有单元必须达到 A 级（80分+）方可视为完成。
  得分低于 80 分的单元需由同一 Agent（或接替 Agent）重新处理。

================================================================================
  三、Agent 完整工作流程（每位 Agent 均按此流程执行）
================================================================================

────────────────────────────────────────────────────────────────────────────────
  输入（Agent 开始前必须阅读）
────────────────────────────────────────────────────────────────────────────────

  1. rule.txt                      — 写作规范全文（尤其是第三节写作规范、
                                     第八节标记语法速查、第九节评分标准）
  2. content/XX-module/unit.txt    — 你负责的单元现有内容
  3. report-template.txt           — 报告填写模板（了解产出格式）
  4. report.txt                    — 了解其他单元的状态（避免重复/冲突）

────────────────────────────────────────────────────────────────────────────────
  执行（四阶段，顺序不可跳过）
────────────────────────────────────────────────────────────────────────────────

  阶段 A — 交互规划
    ▸ 阅读现有 .txt 内容，识别所有知识点
    ▸ 为每个核心知识点规划最合适的交互元素类型和数量
    ▸ 确认总交互元素数量符合密度规范：
        入门级（01-basics）：8-12 个
        中级（02~06）：       6-10 个
        高级（07~18）：       5-8 个
    ▸ 记录「交互元素清单」备用

  阶段 B — 内容重写
    ▸ 按 rule.txt 3.1 节结构模板写作（常规课文 / 项目文件各有模板）
    ▸ 语气严格符合难度等级（入门/中级/高级）
    ▸ 植入阶段 A 规划的交互标记
    ▸ 术语首次出现标注中英双语
    ▸ 无跳步词（"显然""众所周知"等）

  阶段 C — 作业设计
    ▸ 确保有「实战作业」节
    ▸ 设计三级题目：
        🟢 基础题：提供可运行代码框架（\run 块）
        🟡 进阶题：有测试用例或输出参照
        🔴 挑战题：有 \collapse 折叠参考思路
    ▸ 项目文件（project-*.txt）使用「评分标准」节替代作业

  阶段 D — 报告填写
    ▸ 按 report-template.txt 格式填写本单元报告
    ▸ 将报告内容写入 report.txt 对应条目（状态改为 ✅）
    ▸ 将得分追加到 workflow.txt（格式见 workflow.txt 示例）

────────────────────────────────────────────────────────────────────────────────
  输出（Agent 提交时包含以下文件变更）
────────────────────────────────────────────────────────────────────────────────

  必须提交：
    ✦ content/XX-module/unit.txt    — 更新后的单元正文
    ✦ report.txt                    — 对应条目已填写（状态、清单、备注）
    ✦ workflow.txt                  — 追加得分记录

  可选提交（如创建了辅助文件）：
    ✦ 任何新增资源文件（如 SVG、数据文件）放入对应模块目录

  禁止提交：
    ✗ 修改其他单元的 .txt 内容（除非有明确跨单元协作指令）
    ✗ 修改 rule.txt 或 report-template.txt
    ✗ 修改 index.html 或 shared/ 目录

  四、用户查看与后续管理指南
================================================================================

────────────────────────────────────────────────────────────────────────────────
  如何查看 Agent 产出
────────────────────────────────────────────────────────────────────────────────

  1. 查看单元完成状态
     打开 report.txt，检索对应文件名，查看状态标记：
       ⬜ 未开始 | 🔄 进行中 | ✅ 已完成 | ⏸ 暂停/阻塞

  2. 查看单元得分
     打开 workflow.txt，找到对应模块区块，查看详细得分。

  3. 查看详细报告
     report.txt 中每个条目包含交互元素清单、作业设计清单和备注。
     完整的思维链报告（按 report-template.txt 填写）在对应条目的「备注」
     或扩展区域中。

  4. 预览单元内容
     在浏览器中打开 index.html，通过路线图导航到对应章节预览渲染效果。

────────────────────────────────────────────────────────────────────────────────
  质量管理流程
────────────────────────────────────────────────────────────────────────────────

  验收标准：所有单元 ≥ 80 分（A 级）
  
  ┌─────────────────────────────────────────────────────────┐
  │ 得分 ≥ 80 → 标记 ✅，记录到 workflow.txt，关闭任务     │
  │ 得分 70-79 → 标记 🔄，列出改进项，分配修改任务         │
  │ 得分 < 70  → 标记 ⏸，退回重做，重新分配或原 Agent 修改 │
  └─────────────────────────────────────────────────────────┘

  批量状态检查命令（在仓库根目录运行）：
    grep -c "✅" report.txt   # 查看已完成单元数量
    grep -c "⬜" report.txt   # 查看未开始单元数量
    grep -c "⏸" report.txt   # 查看阻塞单元数量

================================================================================
  五、常见问题（Agent FAQ）
================================================================================

  Q: 现有 .txt 内容已经很好，我可以只做小幅修改吗？
  A: 可以。如果现有内容已符合 rule.txt 规范，专注于植入交互标记即可。
     请在报告阶段 B 的「关键决策记录」中说明保留的原因。

  Q: 项目文件（project-*.txt）需要写「知识点详解」节吗？
  A: 不需要。项目文件使用专用模板（rule.txt 3.1节）：
     简介 / 项目需求 / 技术要求 / 实现提示 / 评分标准 / 扩展挑战 / 参考资源。

  Q: 我的交互元素数量超出了密度上限怎么办？
  A: 移除服务最弱的元素（如与知识点关联不紧密的装饰性交互），
     或将部分内容合并到 \tabs 或 \collapse 中。

  Q: 发现 rule.txt 有歧义或盲区怎么办？
  A: 在报告【5】改进意见的「对规范/工具的改进建议」中记录，
     不得自行修改 rule.txt，等待项目负责人统一更新。

  Q: 代码示例无法在本地编译验证怎么办？
  A: 标注"// [需渲染引擎验证]"，在报告维度一中说明，并将维度一得分适当下调。
